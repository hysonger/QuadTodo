## Context

当前 QuadTodo 应用的待办项创建流程需要用户手动点击才能开始输入：
- 点击象限的 "+" 按钮创建待办项后，新待办项以只读模式显示
- 用户需要再次点击待办项才能进入编辑模式
- 按回车创建下一个待办项时，新待办项同样不会自动获得焦点

这增加了不必要的点击操作，影响输入流畅度。

## Goals / Non-Goals

**Goals:**
- 新增待办项（通过按钮或回车）后自动进入编辑模式并聚焦输入框
- 空内容待办项失去焦点时自动删除
- 保持现有拖拽、完成状态切换等功能不受影响

**Non-Goals:**
- 不修改数据持久化机制
- 不改变待办项的数据结构
- 不影响拖拽排序功能

## Decisions

### 1. 使用 `isNew` 标记区分新创建的待办项

**决策**: 在 `Todo` 类型中添加可选的 `isNew?: boolean` 字段，或通过组件 props 传递 `autoFocus` 属性。

**理由**:
- 需要一种机制让 `TodoItem` 知道它是否应该自动进入编辑模式
- `isNew` 标记在保存后可以清除，不影响持久化数据
- 替代方案：使用全局状态管理当前正在编辑的 todo ID，但会增加复杂度

### 2. TodoItem 组件内部管理编辑状态

**决策**: 保持 `TodoItem` 现有的 `isEditing` ref 模式，通过 `watch` 或 `onMounted` 在检测到 `isNew` 标记时自动进入编辑模式。

**理由**:
- 符合现有代码模式，`TodoItem` 已经管理自己的编辑状态
- 聚焦逻辑使用 `nextTick` 确保 DOM 更新后执行
- 最小化对父组件的改动

### 3. 空内容删除逻辑复用现有的 `saveEdit` 处理

**决策**: 复用 `TodoItem` 中已有的空内容删除逻辑（`trim() === ''` 时触发 `delete` 事件）。

**理由**:
- `TodoItem.vue` 已经实现了 `saveEdit` 方法，在内容为空时删除待办项
- 只需确保新创建的待办项初始内容为空字符串即可
- 避免重复实现相同的清理逻辑

### 4. 回车创建下一个待办项时传递象限信息

**决策**: `TodoItem` 在触发 `createNext` 事件时，父组件 `TodoList` 需要知道当前象限类型以创建同象限的新待办项。

**理由**:
- 现有代码已经通过 `createNext` 事件链传递象限信息
- `Quadrant` 组件调用 `handleCreateNext` 时会传入 `quadrant` 参数
- 保持现有事件流不变

## Risks / Trade-offs

**[Risk] 快速连续创建可能导致焦点竞争**
→ 使用 `nextTick` 确保 DOM 更新后再聚焦，Vue 的异步更新机制会队列化处理

**[Risk] 自动聚焦可能与拖拽功能冲突**
→ 新创建的待办项只有在 `isEditing` 为 true 时才显示输入框，拖拽句柄在编辑模式下隐藏，避免冲突

**[Risk] 用户可能意外丢失空待办项**
→ 这是预期行为，符合大多数待办应用的用户体验惯例（如 Apple Reminders、Todoist）

**[Trade-off] `isNew` 标记是运行时状态**
→ 页面刷新后无法恢复"新待办项自动编辑"状态，但这是可接受的，因为刷新后用户通常不会期望继续之前的输入流程
